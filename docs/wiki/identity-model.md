# Identity model

Sky stores everyone in one table, `people`, keyed by a TEXT primary key `id`.
There are two kinds of record ("fiche"), distinguished by whether `auth_sub` is
set. This page explains the two kinds, how a login is resolved to a record, and
the admin/self-service operations that move accounts between records.

See also the shorter [docs/ID-MODEL.md](../ID-MODEL.md), which this page
supersedes.

## The two kinds of record

### Placeholder records (no account)

- `id` = `prenom.nom[.promo][.idx]`: an ASCII, accent-free slug generated by
  `generatePersonId` (`database.ts`). On collision it appends `.promo`, then an
  index `.idx`. The id is stable and never changes after creation.
- Created when a signed-in user adds an entourage member who has no record yet
  (`createPlaceholderPerson`, with `created_by` = the author's id).
- `auth_sub` is `NULL`. No MiGallery photo, no Canari profile.
- Its identity (name/promo) is editable by relatives via the tree editor.

### Account records (linked to Authentik)

- `id` = the Authentik `sub`. Created on first login when no existing record
  matches (`createAuthedPerson`).
- `auth_sub` = `sub`, protected by a partial unique index (one account per
  record). The `sub` is also the key for the MiGallery avatar
  (`/api/avatar/{sub}`) and the Canari profile (`/api/external/profile/{sub}`).
- Identity is owned by MiConnect and refreshed on every login; it is not editable
  from the tree.

## Login resolution

`resolveLogin(identity)` in `database.ts` maps an `OidcIdentity` to a record. It
returns either `{ kind: "linked", personId, created }` or
`{ kind: "choice", candidates }`:

1. **Already linked**: `auth_sub == sub` exists -> `refreshPersonIdentity`, done.
2. **One confident match**: exactly one unlinked record whose normalized last +
   first name match and whose promo matches -> auto-link (`linkPersonAuth`).
3. **Several exact candidates**: return `choice` -> the `/auth/link` screen.
4. **No exact match but fuzzy candidates** (typo / name inversion, via
   `findUnlinkedFuzzyByName`): return `choice` for confirmation. Sky never
   auto-links on a mere resemblance.
5. **No candidate at all**: create an account record (`createAuthedPerson`,
   `id = sub`), returned with `created: true`.

Name comparison uses the tolerant matching in `format.ts` (normalization,
Levenshtein distance, last/first inversion) described in
[matching-and-search.md](matching-and-search.md). Promo matching is exact when
both sides have a promo.

### The disambiguation screen

When `resolveLogin` returns `choice`, the callback mints a **pending-link token**
(`createPendingLink`, stored in the `pending_links` table) and sets the
`__pending_link` cookie (15 min). The token, not the identity, travels with the
client. `/auth/link` shows the candidate stars; the user picks one (link it) or
chooses "create my record". A freshly created record triggers a positions
recompute so the new star appears on the map.

## Moving accounts between records

One account = one person; a record already linked cannot be claimed by another
account. Three operations rearrange links:

- **Relink (self-service)** - `relinkSelf(currentId, targetId, sessionToken)`,
  behind `account/` and `PUT /api/relatives/...` style flows. Moves the signed-in
  user's account from its current record to another **unlinked** record
  (recovering from a wrong auto-link or a homonym). The old record becomes a
  placeholder again **keeping its graph**; the target inherits the account
  identity (sub, name, promo, role); the current session token is repointed so
  the user stays logged in. Because `auth_sub` is UNIQUE, the old record is
  detached first, then the target is linked. Fails if the current record is not
  linked, or the target is the same / missing / already linked.

- **Unlink (admin)** - `unlinkPersonAuth(id)`. Clears `auth_sub`, resets `role`
  to `'user'`, and revokes the record's sessions. The graph and sponsorship links
  are kept; the record becomes a placeholder.

- **Merge (admin)** - `mergePeople(sourceId, targetId)`. Collapses two records
  into one, keeping the sponsorship links. A record linked to an account always
  survives (one star = one person), so two linked accounts cannot be merged
  (`/api/admin/merge` returns an error). Duplicate-detection to suggest merges is
  covered in [sponsorship-graph.md](sponsorship-graph.md).

## Data ownership

Sky is the source of truth **only** for the sponsorship links (`relationships`).
The rest of a profile (bio, current and former clubs) comes from **Canari** via
its public API, keyed by `sub` (see [integrations.md](integrations.md)). The
historical `bio`, `external_links` and `associations` columns/tables are no
longer written by Sky; that data survives in the read-only `sky-legacy.db`
snapshot (see [data-model.md](data-model.md)).
